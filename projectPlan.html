Project Plan

Overview

The Event Generator is a tool based in C# that simulates machine events to test and troubleshoot systems through a web application. Through this tool, users can create, manage,and schedule events the generators need through the user interface. The existing Event Generator forces users to be at each node they want to use. Through this project, current users will be able to access the Event Generator tool through a web application without actually being on their local server. Our team will be meeting with our sponsors and coach weekly on Tuesdays from 6-7 PM. Besides that, the team will hold two meetings per week, separately from our sponsors and coach. These will take place on Tuesdays and Thursdays from 5-6:15 PM, both days. We believe that this frequency of meetings will be enough to keep on track with our project deliverables, and enough to keep everybody updated on each other's progress. If we do have to, we can always increase the number of days or the amount of time we meet accordingly. We plan on following the schedule below to fulfill the requirements of this project. (See Scheduling and Estimates)

Goals and Scope

As for our overall goals for the project, we have been provided both primary and secondary features that consumers have requested. From a high level, one of the most critical features is to centralize the control of generators from a single website. In addition to the website, LenelS2 would like a new DLL that uses C# instead of the outdated C program currently in use. Other features specify the nature of how events should be created, scheduled, and tracked. All primary features will be implemented by the end of the project. Secondary features will only be incorporated after all the primary features are completed and time still remains to work on implementation. Such secondary features include the ability to:

  • import a set of events from the database's events table as a schedule
  • determine the status of generators on a webpage
  • list events that will happen in the future

It is the responsibility of the requirements coordinators on our team to track and maintain the requirements document. Any changes made to the requirements must be agreed upon by both parties after the initial agreement (ex: changing a secondary feature to a primary feature.

  • Multiple people keeping each other in check (what’s reasonable and what is not).

  • Follow safety procedures and guidelines. If a member contracts COVID-19, depending on the severity, each member's workload can be adjusted to compensate for ill team members.
  • Make sure development environments are consistent between team members—document all resolved issues for future reference.
  • Maintain good repo manners. Use git control. NEVER commit to master. Pull requests must be approved by two team members.

Technical Process

  • Methodology

    • For this project, we will be using the Evolutionary Prototyping development methodology. For this methodology, we will be developing incremental prototypes of the system and iterating on each one after receiving feedback from our sponsors. Once we have developed a minimum viable product that the sponsors approve of, then we will move our focus to reaching our stretch goals, if possible, and then eventually finalizing the product.

  • Required Tools and Techniques

    • The system will be developed using C# and the sponsor’s OpenDevice API for the Event Generator and HTML, CSS, and JavaScript for the Web Application. The system event generator, as well as any created services, need to be able to run on the OnGuard server (Windows 8.1 or Windows Server 2016 and later). The Event Generator will also include an installer that is compatible with the outlined technologies previously mentioned.

  • Internal Artifacts Maintained

    • We will be using GitHub and GitHub Issue Tracker to maintain our artifacts, product backlog, and bugs.
